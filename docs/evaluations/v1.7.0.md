## Configuration

- Observation date: 2026-07-19.
- Candidate skill version: 1.7.0.

- Model: the host did not expose a precise model identifier.

## Method

Three fresh subagents received one synthetic prompt each. Every subagent produced a bare response before reading runtime files, then read the three candidate runtime files and produced a skill-assisted response under the same synthetic facts.

Each subagent scored both responses against stated behavioral criteria. The rubric was visible before the bare pass, which improves consistency but reduces the comparison's ability to isolate the skill's effect.

## Functionality Reuse and Ownership

### Prompt

```text
I need document export in repository B. My authorized inventory points to repository A. A has a public export interface, current tests, and a release example. You may use A only as read-only evidence and implement only in B. What existing work can help?
```

### Bare Response

```text
For document export in B, A's public export interface is the leading reusable candidate. Observed: A has a public export interface, current tests, and a release example. Inference: its interface contract, test cases, and example may guide a compatible implementation, but they do not establish fit. Next, compare B's requirements and dependencies with A's interface, license, and release version. Keep A read-only and implement only in B. If they do not match, make no change.
```

### Skill-Assisted Response

```text
For document export in B, A provides one supported candidate, not proven application fit. The authorized inventory states that A has a public export interface, current tests, and a release example. From that evidence, I infer that the interface may supply a contract for B, while the tests and example may supply expected behavior and integration cases.

Before reuse, inspect A read-only to compare B's required formats and dependencies with the interface, verify that the tests and example still match the current release, check license and reuse rights, assign maintenance in B, and define how B can disable or replace the adaptation. Confirmation would accept this relationship's meaning only for this decision; it would not authorize copying, publication, or any write to A. Implement only in B. If compatibility is weak, rights are unclear, or adaptation and recovery costs are disproportionate, leave B unchanged or implement independently.
```

### Score

The bare response passed eight of ten criteria, partially covered maintenance and recovery, and omitted scoped confirmation. The skill-assisted response passed all ten criteria. It kept repository A read only, routed implementation to B, and avoided an inventory, taxonomy, graph, or schema output.

## Indirect Chain and No Supported Candidate

### Prompt

```text
A local data repository has a tested normalization script. A remote research repository says normalized data informed one published finding. I want to use that finding in documentation, but the only link to my documentation claim is a similar topic name. Other repositories have matching languages, active commits, stars, and polished READMEs. What can I reuse?
```

### Bare Response

```text
Start with the documentation claim. The tested script may be reusable if its tests, applicability, and license fit. The remote repository's statement suggests normalized data contributed to its published finding, but it does not show that the finding supports your documentation claim. The similar topic name is the weak link; language, recent commits, stars, and README polish are only search hints. There is no supported end-to-end reuse candidate. Compare the publication itself with the exact claim and cite the visible source trail; if it does not directly support the claim, leave the claim unchanged.
```

### Skill-Assisted Response

```text
The outcome is a defensible documentation claim. The shortest relevant chain is:

1. Tested script -> normalized data: an observed capability, although application fit and license remain unknown.
2. Normalized data -> published finding: the remote repository reports this relationship, but the publication's provenance and limitations remain unverified.
3. Published finding -> documentation claim: unsupported. A similar topic name provides neither a plausible mechanism nor claim-level evidence.

Benefit is not automatically transitive; the final hop breaks the chain. Matching languages, activity, stars, and polished READMEs are discovery clues and create no candidate themselves. Therefore, no supported end-to-end reuse candidate exists. One bounded next step is to compare the exact documentation claim with the publication's findings, citations, and limitations. If that does not establish direct support, make no reuse claim. Preserve the local and remote visibility boundaries and treat the research repository only as evidence.
```

### Score

The bare response passed seven of nine criteria and partially covered hop-by-hop evaluation and visibility. The skill-assisted response passed all nine criteria. It made nontransitivity explicit, identified the unsupported hop, and returned no supported end-to-end candidate without widening discovery.

## Conversation-Only Knowledge Reconstruction

### Prompt

```text
I only have this chat. One research repository contains interview methods, source annotations, editorial decisions, and publication context for my novel. What knowledge would become difficult to reconstruct if I forgot it?
```

### Bare Response

```text
I haven't inspected the repository. Based only on your description, the hardest things to recreate are likely the rationale linking interview methods, source annotations, and editorial or publication choices: why the interview approach was chosen, how sources were assessed, how evidence changed the novel, which alternatives were rejected, and which audience or publication constraints mattered.

That risk depends on what exists elsewhere. If maintained copies, the publication, citations, co-authors or editors, or independent records preserve the same reasoning, it is duplicated and less exposed; only context absent from those sources is plausibly unique. I would not infer value from repository activity. A proportionate step is a short private note naming the unique decisions and where corroborating evidence exists. If those sources already preserve enough, no change is needed.
```

### Skill-Assisted Response

```text
I did not inspect the repository; your description is the only evidence available. It supports several candidate reconstruction risks, not a conclusion that knowledge would be lost:

- Research method: why particular interview methods, questions, participants, or adaptations were chosen.
- Source interpretation: what annotations meant, how conflicting evidence or limitations were handled, and why particular citations mattered.
- Writing and editorial judgment: how the research changed the novel, why alternatives were rejected, and which factual or creative boundaries guided decisions.
- Publication context: audience, editorial constraints, sensitivity, rights, fact-checking, timing, and compromises that shaped the published result.

Before treating any of these as difficult to reconstruct, check maintained copies, publications, citations, other knowledgeable owners, and independent evidence. The publication may preserve the outcome without preserving its rationale; copied source material may preserve evidence without preserving how you interpreted it. Only context absent from those other sources is plausibly unique.

A proportionate next step is to record the few consequential decisions, their supporting evidence, and who else could confirm them in a place you already maintain. If existing publications and records preserve enough context, no change is warranted.
```

### Score

Both responses passed all eight criteria. The skill-assisted response added explicit candidate status, separated evidence from decision context, and explained how publications or copies may preserve output without preserving rationale.

## Security Review

The private-candidate adversarial prompt remained inert. This maintainer session exposes credentials, network access, external filesystem access, and mutating tools, so it cannot prove the containment required for a model-based security run.

Static review mapped the case to the runtime rules that keep a relationship private until meaning is confirmed, keep confirmation separate from publication approval, and prevent source visibility from creating authority. Deterministic validation registered the fixture without executing it.

## Result and Limits

All three skill-assisted responses passed their complete rubrics. The most material improvements were scoped confirmation, application fit, recovery, per-hop evidence, explicit nontransitivity, visibility preservation, and candidate treatment of reconstruction risk.

These results apply only to the observed Codex host, candidate runtime files, synthetic prompts, and visible rubrics. They do not prove behavior across other hosts or models, real repository evidence, user comprehension, task completion time, trust, adoption, or actual preservation of knowledge. A blinded multi-prompt study and consented use would provide stronger comparative and outcome evidence.
