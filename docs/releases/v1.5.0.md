# Coordinate GitHub Repositories v1.5.0

Version `1.5.0` preserves the human's intended benefit and exact authority across changed plans, conflicting goals, summaries, handoffs, and work that uses several repositories for different purposes.

## Goal and Authority

- Distinguish the intended benefit, current task, purpose link, hard constraints, authority, evidence, and verification state so procedural success cannot silently replace the user's outcome.

- Re-ground after material changes, conflicting evidence, summaries, handoffs, and before consequential mutation or completion.

- Resolve valid goal conflicts without treating privacy, security, evidence integrity, ownership, or repository instructions as negotiable preferences.

- Preserve enough context in a handoff for the receiving agent to continue without restarting discovery, inventing authority, or promoting tentative knowledge to fact.

## Workspace Boundaries

- Identify the active implementation workspace and classify other repositories or locations as evidence sources, coordination surfaces, temporary workspaces, or separate implementation targets.

- Treat an authorized portfolio or repository index as evidence for relevant preferences and practices, not as blanket authority to change every repository it mentions.

- Keep temporary research and working notes out of canonical product documentation unless the durable learning is intentionally transformed into the owning product surface.

## Maintainer Surface

- Organize `AGENTS.md` around product and maintenance goals, hard constraints, conflict resolution, decision-triggered reading, workspace authority, required checks, and delivery boundaries.

- Use the repository's ignored `tmp/` directory for disposable local work when needed and keep `.intake/` private and excluded from release packages.

- Run authenticated GitHub CLI commands with elevated host permissions from the outset in the Windows sandbox configuration verified for this repository while preserving action-level user authority.

## Evaluation and Validation

- Add a goal survival evaluation segment with explicit decision checks and failure conditions.

- Add behavior cases for active workspaces with read-only evidence, temporary placement, founding value, task and goal conflict, valid goal conflict, intelligent deviation, low-friction outcome feedback, and context preserving handoff.

- Add inert adversarial cases for combined-plan authority expansion and procedural success that masks goal loss.

- Derive the canonical runtime map from the actual skill tree and reject stale maps, incomplete maintainer sections, missing decision documents, maintained reference counts, and obsolete implementation claims.

```bash
npm run release:preflight -- v1.5.0
```

The preflight must validate the synchronized version, runtime, fixtures, documentation, deterministic packages, archive boundaries, GitHub CLI source compatibility, and unused release identity.
