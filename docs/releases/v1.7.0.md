# Coordinate GitHub Repositories v1.7.0

Version `1.7.0` makes benefit relationships a first-class runtime workflow so an agent can find existing functionality or knowledge that may contribute to a current outcome without requiring a complete inventory or persistent graph.

## Portable Runtime

- Add a focused `benefit-relationships.md` reference that loads for functionality reuse, knowledge reuse, and knowledge reconstruction questions.
- Begin with the current outcome, reuse explicit context and an authorized inventory when available, and inspect only evidence that could change the decision.
- Distinguish an observed capability or body of knowledge from the inferred contribution it may make.
- Keep candidate meaning separate from confirmation for one decision.
- Test each hop in an indirect chain and reject automatic transitivity.
- Compare reuse, adaptation, new work, and no change after checking application fit.
- Route implementation to the owning repository without granting write authority to evidence sources.

## User Output

- Return one to three supported candidates or a no-supported-candidate result by default.
- Explain what may help, how it contributes, the decisive evidence, a material caveat or unknown, and the next bounded verification.
- Surface detailed provenance, observation time, confidence, visibility, ownership, and refresh information only when they affect the decision or the user requests a durable artifact.
- Treat knowledge loss as a reconstruction question and check maintained copies, publications, citations, other owners, and independent evidence before recommending preservation.

## Architecture and Safety

- Record the focused runtime boundary in [Architecture Decision 0004](../decisions/0004-separate-benefit-discovery-from-inventory.md).
- Keep practice reuse and inventory administration in their existing focused reference.
- Reuse an existing inventory, coordination surface, or owner-maintained document only after the user approves persistence, visibility, ownership, and refresh.
- Prevent a private inferred relationship from becoming confirmed portfolio knowledge or public output without separate meaning and publication review.
- Keep the skill conversation capable and independent of a taxonomy, graph, profile, connector, portal, manager, external vocabulary, or runtime script.

## Evaluation

- Expand the deterministic registry from 57 cases across 16 segments to 68 cases across 17 segments.
- Add positive activation for functionality and knowledge reconstruction plus a structural dependency false trigger.
- Add behavior cases for functionality, owner routing, indirect chains, local and remote evidence, non-code reconstruction, confirmation and correction, conversation-only use, and no supported candidate.
- Add an inert adversarial case for private relationship promotion.
- Record three fresh synthetic comparisons in [the v1.7.0 forward evaluation](../evaluations/v1.7.0.md). Every skill-assisted response passed its rubric, while the visible rubric, synthetic inputs, and observed host limit the claim.

## Compatibility

This minor release preserves the ten-stage workflow, conversation-only use, portable frontmatter, existing installation paths, package formats, authority for every target, and the no-taxonomy boundary. It adds one directly linked runtime reference and no dependency, service, persistent store, connector, or script.
