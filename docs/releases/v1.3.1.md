# Coordinate GitHub Repositories v1.3.1

Version `1.3.1` improves the skill's portfolio knowledge reuse behavior and replaces the release pipeline with a smaller, deterministic delivery contract derived from evaluated portfolio evidence.

## Runtime Improvement

- Use an authorized portfolio inventory to locate relevant user preferences, analogous repositories, and practices the user already understands.
- Inspect only the bounded instructions, workflows, tools, and runbooks needed for the current decision.
- Evaluate precedents by risk reduction, work removed, outcome fit, portability, recovery behavior, and maintenance cost instead of copying them.
- Combine compatible controls when the combined design fits better, reject unrelated deployment machinery, and preserve the boundary between evidence access and write authority.
- Keep private repository identities, account relationships, and local paths out of portable guidance and public evidence.

## Release Reliability

- Create byte-reproducible stored ZIP files through dependency-free Node.js code with sorted names, fixed metadata, normalized paths, and fixed permissions.
- Parse and verify ZIP records without extraction, including inventory, staged byte identity, CRC values, checksums, unsafe paths, private local material, and credential patterns.
- Run one final-tree release preflight that snapshots tracked and nonignored untracked files, validates release identity and unused remote state, runs repository and GitHub CLI source checks, builds twice, requires identical checksums, and rejects candidate-tree mutation.
- Let pull request CI own the protected source gate without repeating it on the resulting `main` commit.
- Guard the remote annotated tag and main ancestry before packaging, then install and verify the exact unpublished tag before creating the draft.
- Refuse to overwrite a draft whose title, notes, or publication state differs from the immutable tagged source.
- Serialize preparation and publication per tag without cancelling release state transitions.
- After publication, verify the versionless public install and a contained update from the previous published release in separate ephemeral runner directories.

## Design Boundary

The release design combines useful controls observed in authorized package, application cutover, and deployed service repositories. It keeps exact-tree preflight, deterministic assets, immutable tag state, narrow publication checks, and contained migration verification. It omits production environments, provider credentials, hotfix ancestry, merge queues, and separate abandonment automation because those controls do not currently improve this skill repository's asset-only, low-volume delivery.

This operational evidence comes from the founding builder and maintainer. It demonstrates another intended use of portfolio knowledge but does not establish independent adoption.

## Verification

Run:

```bash
npm run release:preflight -- v1.3.1
```

The command must leave the complete candidate tree unchanged and produce three deterministic ZIP files plus `SHA256SUMS`. The tag workflow must verify the exact `v1.3.1` install and create an equivalent draft. After publication, both the versionless install and the contained `v1.3.0` to `v1.3.1` update must pass.

Release tag: `v1.3.1`.
