# Secure Launch Guide

## Purpose

Prepare a coordinated public introduction without confusing release readiness with proof that a launch, adoption, or recommendation occurred.

## Launch Readiness

Use the latest public release, its checksums and attestations, the [founding case study](CASE-STUDY-FOUNDING-PORTFOLIO.md), the [quickstart](QUICKSTART.md), the [security policy](../SECURITY.md), the [feedback path](FEEDBACK.md), and a named support owner as the proof set.

Do not launch while an affected security, privacy, evaluation, package-integrity, or values review is red. Missing attention or outside feedback does not block preparation, later maintenance, or unrelated product work.

## Reusable Message

Coordinate GitHub repositories across accounts, organizations, and code or non-code work without replacing the user's existing system. The free portable skill starts with the outcome and current organization, compares no change with proportionate options, and requires explicit authority before repository mutation or public output.

Link to the latest release rather than a maintained version number. State that the founding case came from the builder and maintainer, and do not promise universal results.

## Channel Adaptation

For a Show HN post, lead with the repository-sprawl problem, show the founding result and install path, explain what the skill does not replace, and invite concrete observations. For a relevant community, answer that community's specific problem and current posting rules instead of copying a generic launch message. For Product Hunt, LinkedIn, directories, or an awesome list, use a short factual description, link the latest release, disclose the founding evidence relationship, and follow the current submission or contribution rules.

Prepare each message independently and review it at publication time. Do not create accounts, schedule posts, claim an endorsement, or publish through a personal or organization identity without explicit authority for that channel and exact text.

## Channel Check

Before each post, confirm that the audience has the coordination problem, review the channel's current rules, choose one suitable message, verify links, name the person handling replies, and record a correction path. Do not automate posts, evade community rules, or repeat the same message across unrelated communities.

## Response and Correction

Route questions to Discussions, reproducible defects to the bug form, observations from real runs to the skill-run feedback form, proposed changes to the improvement form, and sensitive security findings to the private security route.

If a claim or asset is wrong, correct the public message, link the correction, record the affected release or evidence, and publish a new semantic version when an artifact changes. Never move a published tag or silently replace a release asset.

## Measurement

Record the baseline window, observation window, durable signals, temporary spikes, participant relationship, and limitations in [Program Evidence](PROGRAM-EVIDENCE.md). Attention is not adoption, recruited feedback is not unsolicited, and a release download does not prove a useful outcome.
